# UltiType Spec

## 概要

打鍵/秒 6.0 以上の上級者を主対象にしたタイピング練習アプリ。キーボード表示はありません。本番モードを解放するためには、仮レーティングがA0以上である必要があります。

出題は、すべて短文または長文で行います。単語の出題はありません。

## 初期雛形で作るもの

- React ベースのタイピング練習画面
- WPM / 打鍵/秒 / 正確率 / ミス数 / 打鍵ペースの表示
- 上級者向けに短文ではなく高密度な文章を出題
- 入力中の文字位置、ミス、進捗が視覚的に分かる UI

## 想定ユーザー
- タイピング速度が概ね 6.0 keys/sec 以上
- 基礎練習よりも速度維持、正確性、難文処理を重視するユーザー

## 練習モード（採点基準変更）

ユーザーは、毎日1回5分～30分練習することを前提とします。連続で長時間練習することは推奨しません。時間制であり、1回120秒とします。

### 1. 正確無比

IMEが無効の状態で、正確度が高いと高スコアになります。（正確率^3→正確率^4.5）

### 2. 行雲流水

IMEが無効の状態で、打鍵と次の打鍵の感覚が均等だと高スコアになります。（「均等率」が追加、スコアに乗算）

### 3. 音速打破

IMEが無効の状態で、速度が高いと高スコアになります。（正確率^3→正確率^1.5）

## 本番モード

テスト形式の課題を2種類から選ぶことができます。時間制であり、1回5分または10分とします。

### 1. IMEなし

IMEを無効にし、タイプを行うとスコアを獲得できます。

### 2. IMEあり

IMEを有効にし、各行を課題と一致させるとスコアを獲得できます。各打鍵のスピードは参考に記録されますが、スコアに反映されません。行を入力完了したあと、Enterをして提出する必要があります。

## レーティングシステム

レーティングスコアは、打鍵/秒 * 1000 * (正確率 ^ 3)を計算式とします。

500点未満をランクなし（NR）とし、500点以上をG0ランクとし、その後90点刻みでG1、G2…と上昇します。ランク一覧は以下のとおりです。

NR　灰
G0-G6　灰
F0-F6　薄緑
E0-E6　薄青
D0-D6　緑
C0-C6　青
B0-B6　橙
A0-A6　赤
S0-S6　金
M0-M19　紫
M20-M39　黒紫
M40-M59　黒
UM0～　虹（数字は青天井）

各ランクにはアドバイスがあります。
G：誰でも最初はここから。
F：ホームポジション・ローマ字を意識して。
E：努力の成果が出てきています。正確さを意識しましょう。
D：実用できるまであと少しです。
C：普段からタッチタイピングをしてみましょう。
B：普段遣いなら十分。仕事ならAを目指しましょう。
A：仕事でも十分通用します。
S：周りの人に自慢できるレベルです。
M：修行の始まりです。練習と実用を往復して伸ばしましょう。
UM：アルティメット・タイパーを名乗りましょう。

### 仮レーティング

練習モード限定のレーティングです。

### レーティング

本番モード限定のレーティングです。

## 技術

React（Next.js） + Bun

仮組の間はDBとしてローカルストレージ＋SQLiteを使用する

## 計算詳細

打鍵/秒について…小数第2位まで使います。全打鍵数で計算してください。IMEありモードの場合、必要な打鍵数を概算してください。

正確率…正解文字数/入力文字数　としてください。IMEありモードの場合、正解文字数/課題文字数としてください。

ミスロック…ミスをした場合、以下の扱いとします。
練習モード・正確無比/本番モード・IME有効：その回数Backspaceを押さなければならず、無視して先に進めることはできません。
その他のモード：誤字をしたまま、そのまま先に進めることができます。Backspaceを使う必要はありません。

レーティングの計算：ひとまずは、最高値のみを採用してください。

課題文：適当に生成して内蔵してください。短文50個と長文20個位あれば十分だと思います。当面の間短文は練習モードに、長文は本番モードにのみ使用します。

IMEあり本番モード一致判定：一貫性があれば、「、」と「，」などの表記揺れを許容します。アルファベットは全角、半角、大文字、小文字を一致させなければいけません。

## 今後の実装予定

### 課題追加機能とふりがな自動下書き

ユーザーが課題文を追加できる機能を作るときは、1 行 1 課題の markdown 風注釈形式を保存形式として使います。`reading` と `furigana` の独立フィールドは持たず、表示文内に `[表示](よみ)` を埋め込みます。例: `[結](けっ)[果](か)を[見](み)てから[今日](きょう)は[帰](かえ)る`。記法文字そのものを表示したい場合は `\[`, `\]`, `\(`, `\)`, `\\` のようにエスケープします。

アプリ内部では、この 1 行から `display`、入力用の `reading`、ruby グループを導出します。`[今日](きょう)` のような複数漢字の ruby グループは分割せず保持し、今後の「漢字とふりがなの同時送り」に利用できるようにします。

ふりがな入力の補助として、課題文から読みと漢字ごとのふりがな候補を自動生成する下書き機能を検討します。候補ライブラリは以下です。

- `kuroshiro` + `kuroshiro-analyzer-kuromoji`: 課題文から読み候補を生成する。
- `furigana`: 表記と読みのペアから、漢字ごとのふりがな割り当て候補を作る。
- `JmdictFurigana`: 単語単位の辞書データとして、熟語内の漢字ごとの読み割り当て精度を補助する。

完全自動確定にはしません。人名、地名、当て字、熟字訓、文脈依存の読みで誤り得るため、自動生成結果は「下書き」として扱い、保存前にユーザーが注釈済み表示文を確認・修正できる UI を用意します。
